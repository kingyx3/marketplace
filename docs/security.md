# Security

## Configuration handling

GitHub Environments are the source of truth for deployed configuration. Runtime
values are entered once in the matching GitHub Environment, validated by CI, and
synced to Vercel during deployment. Vercel stores the resulting runtime env, but
it is not hand-edited as the source of truth.

The Supabase runtime keys use the current key names:

| Purpose | Env name | GitHub location |
| --- | --- | --- |
| Browser-safe, RLS-enforced Supabase access | `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` | variable |
| Trusted server-side Supabase access | `SUPABASE_SECRET_KEY` | secret |

`.env` files are generated by `scripts/generate-env.mjs` for local checks or CI
sync work, chmod 600 when written, and gitignored. Validation output names keys
only, never values. Deploy creates a temporary `.env.deploy`, syncs runtime keys
to Vercel, and removes the temporary file before deployment.

## Response headers

`vercel.json` defines production-safe response headers for every route: frame
embedding is denied, MIME sniffing is disabled, referrers are trimmed
cross-origin, browser permissions are minimized, and API routes return
`Cache-Control: no-store, max-age=0`.

## Row-level security

RLS is enabled on every table and Data API exposure is granted explicitly in
migrations. Policy tiers:

1. **Public read** — catalog products/SKUs, availability, published listing
   items, and active storefront configuration only.
2. **Own rows** — customers select their own commercial documents via
   `auth.uid()`; customer profile updates include `WITH CHECK` so a row cannot
   be reassigned to another auth user.
3. **Trusted server only** — supply, pricing, allocation, refunds, audit,
   webhook, payment-exception, and admin mutation paths have no client-write
   policies.

All writes to commercial or admin-managed tables go through trusted server code,
so price calculation, stock checks, state machines, and storefront publication
controls cannot be bypassed from a browser.

## Storage

Product images live in the public Supabase Storage bucket `product-images`,
created by SQL migration. Public reads are allowed for catalog media; object
insert/update/delete is limited to the service role or authenticated active
staff through `current_user_is_staff()`.

## Admin boundary

Admin routes require a server-verified active record in `staff_users`. App
metadata roles may be read for diagnostics, but they do not authorize an admin
API operation. Order/payment mutations are explicit actions, not generic status
writes. Product/SKU, listing/configuration, inventory, wholesale, supplier PO,
and payment reconciliation changes must follow `docs/admin-operations.md` and
require trusted operator access.

## Webhooks and payments

`app/api/webhooks/stripe/route.ts` verifies `STRIPE_WEBHOOK_SECRET` against the
raw request body, records webhook ids for idempotency, and returns 2xx for
verified-but-ignored events so Stripe does not retry forever.

The client never supplies prices, totals, currency, discounts, inventory, or
billing state. The server derives them from database rows and Stripe state, then
checks stored amounts before moving orders or payments forward.

## Notifications

Notification providers are feature-gated by environment keys. Missing provider
configuration records a skipped notification instead of crashing checkout or
webhook processing. Provider tokens, phone-number IDs, and customer contact
targets are not logged by the app.

## Readiness checks

`/api/health` is intentionally shallow. `/api/health?deep=1` checks Supabase
connectivity, Stripe config presence, and notification provider status using key
names/statuses only. It never returns sensitive values.

## Least privilege elsewhere

Workflows request `permissions: contents: read` only. CI jobs for pull requests
need no provider runtime values. Deployment app checks and migration SQL
validation run before mutable Supabase/Vercel deploy jobs.

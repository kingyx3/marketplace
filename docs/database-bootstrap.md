# Database bootstrap test data

The database bootstrap is an idempotent hosted-environment smoke test. It upserts a connected fixture graph across every application-owned table in the `public` schema, exercises the administrator mutation RPCs used by the control centre, and verifies that the final product is visible through anonymous database reads and the deployed storefront.

Supabase-managed `auth` and `storage` tables and private API rate-limit/idempotency ledgers are infrastructure state, not application fixture tables. The workflow uses the Supabase Auth Admin API to provision its two fixture identities but does not directly seed provider-owned tables.

## Run from GitHub Actions

Open **Actions → Bootstrap Database Test Data → Run workflow**, then select `development` or `staging`. Production is intentionally not an option.

The job binds to the selected GitHub Environment and uses the repository's existing Terraform and environment-resolution workflow to obtain that environment's Supabase URL, active publishable and secret keys, and deployed site URL. The selected environment must already be provisioned, migrated, and deployed. Staging also requires `ENABLE_RELEASE_TOPOLOGY=true`.

The workflow needs the same environment inputs as the existing hosted bootstrap, including access to Terraform state, Supabase management, and Vercel. It does not print resolved secret values.

## Run from a trusted shell

```bash
TARGET_ENV=development npm run db:bootstrap -- --target=development
```

The shell must provide or resolve these values for the selected environment:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`
- `SUPABASE_SECRET_KEY`
- `NEXT_PUBLIC_SITE_URL`
- optional `VERCEL_AUTOMATION_BYPASS_SECRET` for protected deployments

The command rejects targets other than `development` or `staging` and rejects production-looking Supabase or site URLs.

## Fixture and verification contract

The command is safe to repeat. Stable IDs and natural conflict keys update the same fixture records instead of creating duplicate fixture graphs. It maintains at least one connected record in every active application-owned public table, including:

- a bootstrap administrator, access grant, and action permissions;
- a customer and connected order, preorder, payments, refund, shipment, notification, waitlist, webhook, and payment-exception records;
- category, released set, product type, product, variant, SKU, active price, sellable inventory, supplier, purchase order, listing, storefront configuration, allocation rule, and active public promotion;
- an explicit completion audit record in addition to audit rows generated by normal mutations.

The catalog fixture is active, priced, stocked above safety stock, configured as `available_now`, and published. The workflow then:

1. updates representative catalog, supplier, listing, promotion, and publication resources through the production administrator RPCs;
2. reads the connected catalog graph with the secret client;
3. reads the product as an anonymous Supabase client to verify RLS and publication behavior;
4. requests the deployed `/products` and `/products/{slug}` pages and fails unless the fixture is rendered.

The script derives the active public-table inventory from ordered migrations and fails when a new table lacks a bootstrap handler or a removed table remains in the manifest. This makes the fixture the base layer for future integration and end-to-end test data rather than a one-off seed file.

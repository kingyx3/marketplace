# Database bootstrap test data

The database bootstrap is an idempotent hosted-environment smoke test. It upserts a connected fixture graph across every application-owned table in the `public` schema, exercises the mutation RPCs used by the application and control centre, and verifies that the final product is visible through anonymous database reads and the deployed storefront.

Supabase-managed `auth` and `storage` tables are provider-owned rather than application fixture tables. The workflow provisions its two fixture identities through the Supabase Auth Admin API. Server-only rate-limit and idempotency tables are included with isolated bootstrap keys so their atomic RPCs are tested without affecting normal application traffic.

## CI/CD execution

The reusable database-bootstrap workflow is called automatically by the deployment pipeline after a successful development or staging deployment. Production deployments never call it.

The deployment workflow passes the immutable deployment URL directly to the bootstrap workflow. The bootstrap then reads the already-provisioned platform Terraform state and resolves the selected environment's Supabase URL, publishable key, and secret key through the existing environment resolver. This avoids maintaining a second copy of application runtime variables and secrets solely for test-data creation.

The bootstrap stage itself consumes only:

- `GCP_TERRAFORM_CREDENTIALS_JSON`, used to read the existing Terraform state;
- `SUPABASE_ACCESS_TOKEN`, used to resolve the active project-scoped API keys;
- optional `VERCEL_AUTOMATION_BYPASS_SECRET` when deployment protection is enabled.

`GCP_PROJECT_ID`, `PROJECT_SLUG`, `TF_STATE_BUCKET_NAME`, and `TF_STATE_BUCKET_LOCATION` remain optional overrides. They are otherwise derived from the Google credential, repository name, and repository conventions. The stage does not require separately configured `APP_NAME`, site URL, Supabase runtime keys, Stripe settings, Google OAuth settings, or Vercel deployment credentials.

A bootstrap failure fails the development or staging deployment workflow before its final smoke check. This makes the data bootstrap and public-product visibility check part of the normal CI/CD release contract.

## Run from a trusted shell

```bash
TARGET_ENV=development npm run db:bootstrap -- --target=development
```

The shell must provide or resolve these values for the selected environment:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`
- `SUPABASE_SECRET_KEY`
- `NEXT_PUBLIC_SITE_URL`
- optional `VERCEL_AUTOMATION_BYPASS_SECRET` for protected deployments

The command rejects targets other than `development` or `staging` and rejects production-looking Supabase or site URLs.

## Fixture and verification contract

The command is safe to repeat. Stable IDs, natural conflict keys, and fixture-scoped cleanup converge on the same connected graph instead of multiplying test data. It maintains at least one record in every active application-owned public table, including:

- a bootstrap administrator, access grant, action permissions, and the migration-owned permission definition reference data;
- a customer and connected order, preorder, payments, refund, shipment, notification, waitlist, webhook, and payment-exception records;
- category, released set, product type, product, variant, product, active price, sellable inventory, supplier, purchase order, listing, storefront configuration, allocation rule, and active public promotion;
- isolated API rate-limit and completed idempotency records;
- an explicit completion audit record in addition to audit rows generated by normal mutations.

The catalog fixture is active, priced, stocked above safety stock, configured as `available_now`, and published. The workflow then:

1. updates catalog, set lifecycle, product publication, product, price, supplier, listing, promotion, and publication resources through production administrator RPCs;
2. consumes an isolated rate-limit bucket and completes an idempotency record through the server-side protection RPCs;
3. reads the connected catalog graph with the secret client;
4. reads the product as an anonymous Supabase client to verify RLS and publication behavior;
5. requests the deployed `/products` and `/products/{slug}` pages and fails unless the fixture is rendered.

The script derives the active public-table inventory from ordered migrations and fails when a new table lacks a bootstrap handler or a removed table remains in the manifest. This makes the fixture the base layer for future integration and end-to-end test data rather than a one-off seed file.
